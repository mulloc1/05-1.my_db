# 기준 7 · 1:N 관계의 도메인 의미 (구두 설명)

> **평가 항목:** 항목 2 · 설계 설명 — 관계의 비즈니스 의미  
> **질문:** 1:N 목록만이 아니라, **실제 도메인에서 무엇을 의미하는지** 예시를 들어 설명할 수 있는가?

---

## 결론

**예 (본 문서로 보완).** 스키마의 3개 1:N은 “카테고리–도서”, “회원–대출”, “도서–대출”이라는 **도서관 운영 규칙**을 코드로 옮긴 것이다. 아래는 각 관계를 **일상 언어 + 구체 행 예시**로 풀어 쓴 설명이다.

---

## 1:N이란 (도메인 관점)

**한쪽(1)** 에 속한 **부모** 하나에, **다른쪽(N)** 에 속한 **자식**이 **여러 개** 붙을 수 있다.

- 부모 행 **하나** → 자식 행 **0개 이상**
- 자식 행 **하나** → 부모 행 **정확히 하나** (FK)

---

## 관계 1: `category` 1 ──< N `book`

### 도메인 의미

> **“하나의 장르(분류) 아래 여러 권의 책이 속한다.”**

한 권의 책은 **동시에 두 장르**에 속하지 않는다고 가정(과제 단순화). 각 `book`은 **정확히 하나의** `category_id`를 가진다.

### 구체 예시 (시드 데이터)

| category (1) | book (N) |
|--------------|----------|
| Fiction (`id=1`) | `Dune`, `1984`, `To Kill a Mockingbird`, … |
| Technology (`id=5`) | `Clean Code`, `The Pragmatic Programmer` |
| Science (`id=3`) | `A Brief History of Time`, `The Selfish Gene` |

**구두 예:** “Fiction 분류에는 소설 여러 권이 묶여 있고, `1984`는 Fiction **한 곳**에만 등록됩니다.”

**쿼리 연결:** #6 “책 + 장르명”, #9 “장르별 책 수”.

---

## 관계 2: `member` 1 ──< N `rental`

### 도메인 의미

> **“한 회원은 여러 번 대출할 수 있다. 각 대출 기록은 한 회원에게만 귀속된다.”**

회원 카드(Alice) 하나로 3월·6월·9월에 각각 다른 책을 빌리면 **rental 3행**, 모두 `member_id = Alice.id`.

### 구체 예시

| member (1) | rental (N) |
|------------|------------|
| Alice Kim (`id=1`) | 2024-01-10 `Clean Code` 대출 |
| | 2024-04-02 `Sapiens` 대출 |
| | … (시드에 다수) |
| Jordan Yoon (`id=12`) | **(없음 — 0건)** |

**구두 예:** “Alice는 도서관 회원 **한 명**이지만, 대출 **이력**은 여러 줄입니다. Jordan은 회원이지만 아직 한 번도 빌리지 않아 rental 행이 0개입니다.”

**쿼리 연결:** #1 회원 목록, #7 “회원별 대출 횟수(0 포함)”, #10 “2번 이상 빌린 회원”.

---

## 관계 3: `book` 1 ──< N `rental`

### 도메인 의미

> **“한 권의 책(카탈로그 항목)은 시간이 지나며 여러 번 대출될 수 있다.”**

물리적으로 사본 1권만 있다고 해도, **반납 후 재대출**하면 `rental` 행이 또 생긴다. `book`은 **카탈로그 identity**, `rental`은 **대출 사건**.

### 구체 예시

| book (1) | rental (N) |
|----------|------------|
| `Clean Code` (`id=1`) | Alice 2023-05-01 ~ 반납 |
| | Bob 2024-02-14 ~ 반납 |
| | Carol 2024-11-01 ~ **미반납** (`returned_at NULL`) |

**구두 예:** “`Clean Code`는 카탈로그상 **한 항목**인데, 세 명이 **다른 시점**에 빌려 rental이 세 건입니다. 지금은 Carol 것만 아직 안 돌려왔습니다.”

**쿼리 연결:** #3 “최근 대출된 책 TOP 5”, #8 “현재 미반납 목록”.

---

## `rental`이 두 1:N의 교차점

```
        member (1)
            │
            │ member_id
            ▼
         rental (N) ◄── book_id ── book (1)
                              │
                              └── category_id ── category (1)
```

`rental` **한 행**은 동시에:

- **한 회원**의 행동
- **한 도서**에 대한 이벤트

를 나타낸다. **M:N(회원↔책)** 을 풀기 위해 **연결(associative) 테이블** 역할을 `rental`이 겸한다. 추가 컬럼(`rented_at`, `due_at`)으로 “단순 연결” 이상의 **시간 정보**를 담는다.

## FK가 도메인 규칙을 강제하는 예

| 시도 | 도메인에서 말이 되나? | DB |
|------|----------------------|-----|
| 존재하지 않는 회원 id로 대출 | ❌ 가입하지 않은 사람 | FK failed |
| 존재하지 않는 책 id로 대출 | ❌ 카탈로그에 없는 책 | FK failed |
| category 삭제 (책이 남음) | ❌ 분류만 지우고 책은? | RESTRICT |

---

## 1:N vs 1:1 vs M:N (확장 개념)

| 관계 | 도서관 예 | 이 과제 |
|------|----------|---------|
| 1:1 | 회원 ↔ 전자카드 1장 | 미사용 |
| 1:N | 회원 → 대출 여러 건 | ✅ 3곳 |
| M:N | 회원 ↔ 책 (다대다) | `rental`로 **분해** |

JPA/ORM 학습 전 단계에서 “M:N은 **중간 테이블**로 푼다”는 패턴을 SQL로 경험한다.

---

## 구두 설명용 1분 스크립트

> “세 가지 1:N이 있습니다. 첫째, **장르 하나에 책 여러 권**. 둘째, **회원 한 명이 대출 여러 번**. 셋째, **책 한 권이 여러 번 대출될 수 있음**. rental 테이블이 회원과 책을 FK로 묶는 **대출 이벤트**이고, Jordan처럼 대출 0번인 회원도 member에는 있지만 rental에는 행이 없습니다.”
