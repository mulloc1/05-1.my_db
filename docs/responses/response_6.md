# 기준 6 · 테이블 분리 설계 이유 및 각 테이블 역할

> **평가 항목:** 항목 2 · 설계 설명 — 정규화·역할  
> **질문:** ERD에 테이블 관계는 있으나, **왜 이렇게 나눴는지**, **각 테이블의 구체적 역할**을 설명할 수 있는가?

---

## 결론

**예 (본 문서로 보완).** README에는 ERD와 1:N 목록만 있었고 **설계 의도**가 부족했다. 아래는 도서 대출 도메인에서 4테이블을 선택한 **정규화·책임 분리** 근거와 각 테이블의 **단일 책임(SRP)** 이다.

---

## 한 장 요약: 왜 4테이블인가?

```
[마스터/엔티티]          [트랜잭션/이벤트]
 category ──< book ──< rental >── member
  (분류)      (자산)    (대출)     (사람)
```

| 설계 질문 | 답 |
|----------|-----|
| 왜 한 테이블에 안 넣나? | 회원·도서·대출은 **변경 주기·의미·카디널리티**가 다름 |
| 왜 `rental`을 분리하나? | “누가 언제 무엇을 빌렸는지”는 **반복되는 이벤트** — 카탈로그와 분리 |
| 왜 `category`를 분리하나? | 분류명 변경 시 **한 곳만** 수정 (정규화) |

과제 최소 4테이블을 **의미 있는 4역할**로 채웠다: 분류 · 카탈로그 · 회원 · 대출이력.

---

## 각 테이블의 구체적 역할

### 1. `category` — 분류 마스터 (Lookup / Reference)

| 속성 | 설명 |
|------|------|
| **저장하는 것** | Fiction, Science 등 **장르·분류 체계** |
| **저장하지 않는 것** | 개별 도서, 대출, 회원 |
| **변경 빈도** | 낮음 (신규 장르 추가 정도) |
| **대표 컬럼** | `name UNIQUE`, `description` |

**역할:** `book`이 “어떤 종류의 책인가”를 FK 하나로 참조. 분류명을 15권 도서 행마다 문자열로 복사하지 않는다.

**비즈니스 예:** “Technology”를 “Tech & Computing”으로 개명 → `category` **1행 UPDATE**로 5권 도서 분류 표시 일괄 반영 (JOIN 시).

---

### 2. `book` — 도서 카탈로그 (Catalog / Asset)

| 속성 | 설명 |
|------|------|
| **저장하는 것** | 제목, 저자, ISBN, 출판연도, **소속 category** |
| **저장하지 않는 것** | 누가 빌렸는지, 대출 일시 |
| **변경 빈도** | 중간 (신간 입고, 메타 수정) |
| **대표 컬럼** | `title`, `author`, `isbn UNIQUE`, `category_id FK` |

**역할:** **대출 가능한 자산(책)** 의 정적 속성. 한 권의 `Dune`은 `rental`에 **여러 번** 등장할 수 있다(다른 회원·다른 시점).

**비즈니스 예:** ISBN으로 중복 입고 방지(`UNIQUE`). “2015년 이후 출간 도서”는 `book`만 스캔하면 됨 (쿼리 #2).

---

### 3. `member` — 회원 (Party / Actor)

| 속성 | 설명 |
|------|------|
| **저장하는 things** | 이름, 이메일, 연락처, 가입일 |
| **저장하지 않는 것** | 빌린 책 목록(열로 펼치지 않음) |
| **변경 빈도** | 중간 (연락처 변경) |
| **대표 컬럼** | `email UNIQUE`, `joined_at` |

**역할:** **대출 행위의 주체**. 회원 1명 ↔ 대출 N건은 1:N이므로 대출 목록은 `rental`에만 둔다.

**비즈니스 예:** `member#12`는 가입했지만 아직 대출 0건 — `member` 행은 존재하고 `rental`에 FK 없음 (쿼리 #7 LEFT JOIN).

---

### 4. `rental` — 대출 트랜잭션 (Fact / Event)

| 속성 | 설명 |
|------|------|
| **저장하는 것** | **누가**(`member_id`), **무엇을**(`book_id`), **언제 빌렸/반납** |
| **저장하지 않는 것** | 회원 이름·책 제목(중복 저장 안 함 → JOIN) |
| **변경 빈도** | **높음** (대출·반납마다 INSERT/UPDATE) |
| **상태 표현** | `returned_at IS NULL` = **현재 대출 중** |

**역할:** 도메인의 **핵심 가변 데이터**. 거의 모든 흥미로운 쿼리(#3, #5, #8, #10–#12)가 이 테이블에서 출발한다.

**비즈니스 예:** Alice가 `Clean Code`를 2024-03-01에 빌림 → `rental` 1행. 반납 시 `returned_at` UPDATE (쿼리 #13). 별도 `status` ENUM 없이 NULL로 “미반납” 표현 — 컬럼 수 최소화(YAGNI).

---

## 테이블을 나누지 않았을 때의 문제 (반례)

### 단일 거대 시트 (엑셀식)

| member_name | book_title | category | rented_at | … |
|-------------|------------|----------|-----------|---|
| Alice | Clean Code | Technology | 2024-01-01 | … |
| Alice | Clean Code | Technology | 2024-06-01 | … |

| 문제 | 결과 |
|------|------|
| `Clean Code` 메타(저자) 변경 | 모든 rental 행 수정 |
| `Technology` → 이름 변경 | 모든 해당 행 수정 |
| Alice 이메일 변경 | Alice가 빌린 **모든 행** 수정 |
| “현재 카탈로그 도서 수” | rental 행까지 섞여 **중복 카운트** |

**정규화된 4테이블**은 “변경이 일어나는 **한 곳**”만 고친다.

## 설계 결정과 과제 요구의 정렬

| 과제 요구 | 이 스키마의 대응 |
|----------|-----------------|
| ≥ 4 테이블 | category, book, member, rental |
| ≥ 2개 1:N | category→book, member→rental, book→rental (**3개**) |
| PK/FK | 모든 테이블 PK; 3 FK |
| 현실적 seed | rental 25건으로 “도서관 운영” 시뮬레이션 |

`docs/plan.md` §4.2 Table Shapes와 동일한 의도를 README에도 반영하는 것이 기준 6의 목표다.

## 구두 설명용 30초 스크립트

> “도서관 DB는 **분류(category)**, **책(book)**, **회원(member)**, **대출 기록(rental)** 네 덩어리로 나눴습니다. 책 정보와 대출 이력은 바뀌는 속도가 다르기 때문에 분리했고, rental은 member와 book을 FK로 연결하는 **이벤트 테이블**입니다. 미반납은 `returned_at`이 NULL인 행으로 표현합니다.”
